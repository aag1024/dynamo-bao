The token-hash
—bucket index

A probabilistic alternative to full-table scans for substring-style queries in DynamoDB. Instead of one index entry per token, each document writes one entry per bucket — a hash of its tokens, modulo K. Reads target a single partition. False positives are tunable. Write amplification is bounded.

Updates as K and the query change. Same dataset, same semantics — different cost shapes.

A · iter index + searchText projection

Bucketed scan, server-side filter

One record per doc with searchText projected. Query every iter bucket in parallel; DynamoDB applies contains() server-side. Returns exact docIds — no hydration needed.

B · inverted index · KEYS_ONLY

Direct partition lookup per term

One record per (token, doc). One Query per term returns matching docIds directly — every record returned is a match. Multi-term AND/OR is N queries combined client-side with set ops.

C · token-hash-bucket + searchText projection

Hashed partition, server-side verify

One record per (bucket, doc), with searchText projected. Hash narrows the partition; DynamoDB's contains() filter eliminates hash collisions server-side. Returns exact docIds.

Index record · INCLUDE searchText pk = "[acme]#bucket:" + (djb2(token) mod K)
sk = docId
project = searchText Read: hash term → bucket B → Query partition with FilterExpression contains(searchText, term). DynamoDB scans the bucket, filters server-side, returns exact matches. Caps both partition heat (via K) and read scope (via bucketing).

All three approaches return exact docIds with zero false positives. The differences live in items read (RCU cost), storage bytes (capacity cost), write amplification (WCU cost on save), and client complexity (B requires multi-query set operations for compound terms).

On write amp: the record-count metric tells only half the story. B writes many records but each is just keys (~50 bytes) — cheap per write. C writes fewer records than B but each duplicates the full searchText projection — so bytes/save can dominate even when record-count looks competitive. Both bytes and record count contribute to WCU billing (DynamoDB rounds each write up to the nearest KB).
